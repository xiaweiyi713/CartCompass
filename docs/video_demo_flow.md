## 录制前开机清单

录制前先完成，不要在正式录制里临场排错。

### 1. 关闭旧后端

如果 8000 端口有旧实例，先关掉：

```bash
lsof -nP -iTCP:8000 -sTCP:LISTEN
kill <PID>
```

### 2. 激活环境并自检

```bash
cd "/Users/xuwenyao/字节AI全栈挑战赛"
source server/.venv/bin/activate
python --version
PYTHONPATH=server python server/scripts/self_check.py
```

确认：

- Python 是 3.10+，推荐 3.11。
- 数据库商品数是 321。
- 没有缺依赖。

### 3. 用 Chroma 启动后端

正式演示建议显式开 Chroma：

```bash
cd "/Users/xuwenyao/字节AI全栈挑战赛"
source server/.venv/bin/activate
export VECTOR_STORE_BACKEND=chroma
export CHROMA_PATH=server/storage/chroma
PYTHONPATH=server python -m uvicorn app.main:app --host 127.0.0.1 --port 8000
```

另开终端验证：

```bash
source server/.venv/bin/activate
PYTHONPATH=server python server/scripts/self_check.py --require-server
curl -s http://127.0.0.1:8000/api/health | python -m json.tool | sed -n '1,80p'
```

必须能看到类似：

```json
"vector_store": {
  "configured_backend": "chroma",
  "active_backend": "chroma_text_embedding",
  "vector_kind": "text_embedding"
}
```

如果没有文本 embedding，也可能是 `chroma_hashing_vector`。重点是 `configured_backend=chroma` 且 `/admin/metrics` 能显示当前向量库。

### 4. 打开 Dashboard

浏览器打开：

```text
http://127.0.0.1:8000/admin/metrics
```

录制前刷新一次，确认有：

- 商品总数。
- 当前向量库。
- Chroma 持久化。
- 首 token / 检索缓存 / 延迟卡片。
- 最近 Trace 表格。

### 5. iOS 使用真机 + Release

推荐：

- 用真机录屏，Release 构建。
- 后端地址保持 `http://127.0.0.1:8000` 时，模拟器最方便；真机需要确保后端地址可被手机访问，或按工程配置改为 Mac 局域网 IP。
- 如果来不及配真机网络，可以用模拟器录功能，再用真机补拍 UI 流畅片段。正式口播里说明演示推荐真机 Release。

### 6. 准备测试图片

拍照找货使用以下本地商品图更稳：

```text
server/static/product_images/p_anker_001_fc881685.jpg
server/static/product_images/p_real_100_008943c0.jpg
server/static/product_images/p_digital_016.jpg
server/static/product_images/p_clothes_014.jpg
```

如果某个文件名本机不存在，用 `server/static/product_images/` 下同类商品图替代。

### 7. 清理演示状态

录制前做一轮：

- App 侧栏点击“新对话”。
- 如需展示长期偏好，从空白状态开始；如需清空，进入“我的偏好”清除。
- 浏览器 Dashboard 刷新。
- 终端字号放大，避免录屏看不清。

## 逐镜头演示流程：6 分钟版

### 镜头 1：开场定位

画面：README 标题或一张简单标题页。

口播：

> 大家好，这是我的字节 AI 全栈挑战赛项目 ShopGuide。它是一个基于 RAG 的多模态电商导购 AI Agent，前端是原生 iOS SwiftUI，后端是 FastAPI。用户可以像和豆包聊天一样，用文字、语音或图片表达购物需求；系统会基于本地商品库做推荐、对比、追问、加购和下单。核心原则是：商品事实只来自本地数据库和可追溯来源，模型不能编造价格、库存、优惠或平台承诺。

画面可切 README 架构图。

口播：

> 整体链路是 iOS 通过 SSE 连接 FastAPI，后端 Agent 先判断本轮是闲聊、导购、追问、购物车还是多模态搜索；真正的商品检索走 SQL 预过滤、BM25、Chroma 向量库和可信度重排；最后回答通过 GroundingGuard 校验后才流式返回。

### 镜头 2：后端启动与 Chroma 证明

画面：终端执行或展示已经启动的终端。

推荐操作：

```bash
source server/.venv/bin/activate
export VECTOR_STORE_BACKEND=chroma
PYTHONPATH=server python server/scripts/self_check.py --require-server
```

口播：

> 这里先展示工程可运行性。项目提供了 self-check 脚本，会检查 Python 版本、依赖、数据库和健康接口，避免评测时大量手动配置。

画面：浏览器打开 `/api/health` 或 `/admin/metrics`。

口播：

> 这次演示我显式启用了 Chroma。这里可以看到当前向量库 backend 是 Chroma，并且商品库有 321 条数据。Chroma 不是写在代码里的摆设，而是实际参与了向量检索；如果 Chroma 不可用，系统会自动回退到 SQLite hashing 向量，保证 demo 不会因为一个可选依赖失败而跑不起来。

### 镜头 3：模糊需求主动澄清

画面：iOS App 主聊天页。

输入：

```text
推荐手机
```

预期表现：

- Agent 先追问拍照、续航、游戏性能、预算等。
- 不直接胡乱弹商品。

口播：

> 用户只说“推荐手机”时，系统不会为了展示商品而硬推。Agent 会识别这是弱约束购物意图，先追问关键偏好和预算，这样更像真实导购，而不是关键词搜索。

继续输入：

```text
游戏，预算9999
```

预期表现：

- 出现手机商品卡。
- 文本流式输出。
- 商品卡可点击。

口播：

> 用户补充游戏和预算后，系统继承上一轮手机上下文，进入推荐链路。商品卡先于完整文案返回，所以首屏响应很快；文案继续通过 SSE token 流式输出。

### 镜头 4：强约束与反选过滤

输入：

```text
推荐适合油皮的防晒，200元以内，不要含酒精
```

预期表现：

- 切换到美妆护肤，不沿用手机上下文。
- 返回符合预算、防晒、油皮、不含酒精约束的商品。
- 商品卡显示匹配分和命中理由。

口播：

> 这个请求同时包含类目、肤质、预算和反选条件。后端先解析结构化约束，然后在 SQL 层做硬过滤，再进入 BM25 和向量相似度重排。特别是“不含酒精”这种反选条件，是硬约束，不允许后面的向量召回再把被排除商品带回来。

画面：点击第一张商品卡进入详情。

口播：

> 商品详情页展示的不只是图片和价格，还有推荐决策：匹配分、命中依据、风险提示、SKU、评论和公开来源。这是为了让推荐结果可解释、可审计。

### 镜头 5：商品级追问

回到聊天，输入：

```text
第一款差评主要说什么
```

预期表现：

- 不重新推荐。
- 围绕上一轮第一款商品回答。
- 提到评论或风险。

口播：

> 多轮追问依赖会话状态里的 `last_product_ids`。用户说“第一款”时，Agent 能解析指代，并读取对应商品的评论、FAQ、详情 chunk，而不是重新跑一次普通推荐。

继续输入：

```text
这款适合敏感肌吗，有没有酒精
```

口播：

> 成分和适用性问题同样会回到商品库证据。如果库里没有依据，系统会保守说明边界，而不是用模型常识编造。

### 镜头 6：反馈闭环和平替

输入：

```text
第一款太贵了，有没有平替
```

预期表现：

- 返回更便宜替代品。
- 保留同类目或核心偏好。

口播：

> 导购不是一次性推荐。用户反馈“太贵了”后，系统会进入替代品模式，用当前商品作为参照，保留类目和关键需求，重新找更低价候选。这是一个简单但实用的反馈闭环。

### 镜头 7：购物车和下单闭环

操作：

1. 在商品卡或详情页点击加购。
2. 打开购物车看商品、数量、价格。
3. 回聊天输入：

```text
我要下单
```

4. 输入地址：

```text
北京市朝阳区 Demo 路 1 号
```

5. 输入：

```text
确认下单
```

预期表现：

- Agent 汇总订单。
- 需要用户确认。
- 返回订单卡。
- 订单后推荐配件/复购商品。

口播：

> 购物车和下单也是确定性工具，不靠 LLM 编造。加购会校验商品和 SKU，数量变化会更新总价，确认下单前会校验 mock 库存。订单创建后会发出结构化 order 事件，iOS 渲染成订单卡，同时给出订单后推荐。

### 镜头 8：长期偏好记忆

输入：

```text
记住我以后护肤品不要含酒精，我是油皮，预算200
```

预期表现：

- 回复已记住长期偏好。
- profile 事件更新。

继续输入：

```text
推荐防晒
```

口播：

> 长期偏好会保存到后端用户画像。之后用户只说“推荐防晒”，系统会自动带上油皮、预算 200、不含酒精这些条件。iOS 侧也提供偏好查看和清除，避免黑盒记忆。

### 镜头 9：多模态拍照找货

操作：

1. 点击图片按钮。
2. 上传一张手机/防晒/充电器图片。
3. 可在输入框先写 `油皮防晒` 或 `快充充电器` 再上传图片。

口播：

> 多模态部分不是简单的图片分类。上传图片后，后端会用多模态 embedding 把图片放到图文共享向量空间，与商品文本向量做相似度匹配；同时融合 VLM 图像理解和轻量视觉特征。如果用户还输入了文字，系统会把文本约束一起加入排序。

预期表现：

- 返回同类商品。
- 推荐理由包含图片理解或语义匹配。
- Trace 中有 image_search 链路。

### 镜头 10：语音输入与朗读

操作：

1. 点麦克风。
2. 说：“推荐适合通勤的降噪耳机，预算两千以内。”
3. 发送后点朗读按钮。
4. 可展示语速/音色设置。

口播：

> iOS 端还集成了原生语音识别和 TTS。用户可以直接说购物需求，系统转写后进入同一套 Agent/RAG 链路；回答也可以朗读，适合移动端真实使用场景。

### 镜头 11：可观测性和 Trace

画面：浏览器 `/admin/metrics`。

口播：

> 最后展示工程可观测性。Dashboard 汇总了商品覆盖、当前向量库、首 token 延迟、检索延迟、缓存命中率和最近 Trace。每个聊天 done 事件都会带 trace_id。

画面：打开某个 `/api/traces/{trace_id}`。

口播：

> Trace 可以看到本轮从意图识别、约束解析、候选过滤、检索栈、返回商品到 Guard 校验的完整路径。答辩时如果评委问“为什么推荐这款”，可以从商品卡、Trace 和代码三层解释。

### 镜头 12：结尾总结

口播：

> 总结一下，ShopGuide 覆盖了题目要求的基础导购、多轮对话、购物车下单和多模态输入；工程上有原生 iOS 客户端、FastAPI 后端、Chroma 向量库、SQLite 事实库、可观测 Dashboard、自动化测试和 runbook；可靠性上通过 GroundingGuard、结构化工具和 fallback 保证商品事实不幻觉。它不是一个纯 Web Demo，而是一个可以在移动端完整跑通的多模态电商导购 Agent。

## 录制注意事项

- 终端字号放大到 16-18。
- 浏览器只保留 `/admin/metrics`、`/api/health`、Trace 三个标签页。
- iOS 录屏时先打开勿扰模式。
- 如果用模拟器，窗口大小固定，不要频繁缩放。
- 录制时每个输入之间停 1-2 秒，给后期剪辑留空间。
- 展示 Dashboard 时，不要露出 `.env` 或真实 API key。
- 如果 LLM 响应慢，可以强调商品卡已经先返回，这是 fast-path 设计的一部分。
- 如果 Chroma 显示 `chroma_hashing_vector`，可以说当前没有真实 embedding 或没有预计算文本向量时，Chroma 承载本地 hashing 向量；配置 embedding 后会显示 `chroma_text_embedding`。
- 如果某个拍照样例效果不理想，换用 `server/evaluation/cases/multimodal_demo_samples.json` 里已验证的图片。

## 后期剪辑建议

- 开头 5 秒放最终 App 效果，不要一上来全是终端。

- 结尾放 `/admin/metrics` 和 iOS 商品详情页，强化工程质量和可解释性。
